## 페르소나

**수학/통계 기반 연구자** — 1등을 최종 목표로 하되, 1회 플로우 완료 시점에 "당일 최선값"을 반드시 반영합니다.

# 04. Offset 최적화 연구 (재설계)

사용자가 `당첨 이력.md`의 세트별 현황을 보고 "세트#N 수정" 또는 "세트 2,4,5 개선"처럼 다중 세트 개선을 지시하면 이 워크플로우를 실행합니다.

## 핵심 원칙

1. **최종 목표는 1등**
2. **현실 목표는 1회 플로우 완료 시 최선값 반영**
3. **각 방향은 독립 축을 바꿔야 함** (같은 축 반복 금지)
4. **Step 8 전에는 코드/이력 반영 금지**
5. **이력 파일 보호**: Step 8 전 비교 실험에서는 `당첨 이력.md`를 절대 덮어쓰지 않는다.

## 필수 사전 점검

- `jl_service.py`의 `_speed_from_offset()`이 전달된 offset을 실제 speed에 반영하는지 먼저 확인한다.
- 이 조건이 깨지면 offset 탐색 결과는 무효다.

## 파일 범위

| 구분 | 파일 경로 |
|------|----------|
| 핵심 알고리즘 | `features/analysis/api/jl_service.py` |
| 시뮬레이션 | `features/analysis/scripts/run_wheel_52.py` |
| 시드 검증 | `features/analysis/scripts/run_multi_seed.py` |
| 시뮬 결과 | `features/analysis/scripts/당첨 이력.md` |
| 연구 결과 | `features/analysis/scripts/연구 분석 결과.md` |

---

## 3대 연구 방향 (독립 축)

```
[사용자 입력] "세트#N 미당첨, 수정해줘"
       │
       ▼
┌────────────────────────────────────────────┐
│ 방향 1: Offset 탐색                        │
│ - 대상 세트 offset 0~44 탐색              │
│ - 기준: 등수 + 평균 매치 수                │
└──────────────────────┬─────────────────────┘
                       ▼
┌────────────────────────────────────────────┐
│ 방향 2: 시작번호 전략 탐색                 │
│ - previous_draw / blended / global_top6    │
│ - 방향1 최적값 위에서 비교                 │
└──────────────────────┬─────────────────────┘
                       ▼
┌────────────────────────────────────────────┐
│ 방향 3: 구조 탐색                          │
│ - WHEEL_OFFSET_STEPS 패턴 변경             │
│ - 방향1+2 최적값 위에서 비교               │
└──────────────────────┬─────────────────────┘
                       ▼
          [최선값 확정 + 이력/문서 반영]
```

> 중요: 모든 평가는 `--independent-wheels`를 포함한다.

---

## 워크플로우 (1회 실행 완료형)

### Step 1. Baseline 기록

1. `당첨 이력.md`에서 현재 상태 기록
   - 1~4등 합계, 5등 합계, 당첨 누적 회차
2. 52회 baseline 실행 (비교 기준 확보용)

```bash
python -m features.analysis.scripts.run_wheel_52 --seed 42 --independent-wheels --no-history
```

3. baseline의 `평균 매치 수`를 함께 기록한다.
4. **중요**: Step 1은 비교 기준 수집 단계다. `당첨 이력.md` 저장은 금지하고 `--no-history`를 사용한다.

### Step 1.5. 타깃 세트 확정 (세트 2,4,5 우선)

- 기본 타깃 묶음: **세트 2, 4, 5**
- 실행 순서: **2 → 4 → 5** (각 세트 완료 후 다음 세트로 진행)
- 우선순위 규칙:
  1. `당첨 이력.md` 기준 최근 52회 미당첨 세트 우선
  2. 미당첨 동률이면 현재 offset이 이웃 세트와 충돌(거리 0~1)인 세트 우선
  3. 여전히 동률이면 세트 번호가 작은 순서 우선
- 세트별로 Step 2~5를 독립 실행하되, Step 8 전까지는 **임시 best-so-far(메모리/문서)** 로만 관리한다.

### Step 2. 방향 1 — Offset 탐색 (대상 세트 루프)

- 대상 세트 목록(예: `2,4,5`)을 순회하며 세트별로 offset 후보(기본: 0~44 전체) 평가.
- 후보 평가는 아래를 함께 본다:
  - 등수 지표 (3/4/5등)
  - **평균 매치 수**
  - 이웃 세트 offset 충돌 거리

```bash
python -m features.analysis.scripts.run_wheel_52 --seed 42 --independent-wheels --refine-offset 2 --offset-full --no-history
python -m features.analysis.scripts.run_wheel_52 --seed 42 --independent-wheels --refine-offset 4 --offset-full --no-history
python -m features.analysis.scripts.run_wheel_52 --seed 42 --independent-wheels --refine-offset 5 --offset-full --no-history
```

- 각 세트에서 개선 후보가 있으면 **즉시 반영하지 말고**, 임시 최선값(best-so-far 후보)으로만 기록한다.
- 게이트: 특정 세트 개선으로 다른 세트의 4등 이상 합계가 감소하면 해당 후보는 보류하고 차순위 후보를 검토한다.

### Step 3. 방향 2 — 시작번호 전략 탐색 (세트별 best-so-far 고정)

- 방향 1 최선값을 고정한 상태에서 시작번호 전략 비교:
  - `previous_draw`
  - `blended`
  - `global_top6_fixed`

```bash
python -m features.analysis.scripts.run_wheel_52 --seed 42 --independent-wheels --start-strategy previous_draw --no-history
python -m features.analysis.scripts.run_wheel_52 --seed 42 --independent-wheels --start-strategy blended --no-history
python -m features.analysis.scripts.run_wheel_52 --seed 42 --independent-wheels --start-strategy global_top6_fixed --no-history
```

- 세트별 비교 결과를 임시 best-so-far 후보로 갱신 후 다음 타깃 세트에 동일 절차를 적용한다.
- `--no-history` 없이 전략 비교를 실행하면 마지막 실행 결과로 `당첨 이력.md`가 덮어써지므로 금지한다.

### Step 4. 방향 3 — 구조 탐색 (`WHEEL_OFFSET_STEPS`)

- `jl_service.py`의 `WHEEL_OFFSET_STEPS`를 후보 패턴으로 변경해 비교.
- 예: `[0,1,2,4,7,11]`, `[0,2,5,9,14,20]` 등.
- 각 후보는 52회 평가 후 best-so-far와 비교.

### Step 5. 필터 필요성 검증 (A/B, 세트 묶음 최종값 기준)

- 최종 best-so-far 설정에서 필터 ON/OFF 비교:
  - `--combo-filter` ON (`--no-history`)
  - `--combo-filter` OFF (`--no-history`)
- 목적: 필터가 세트 2/4/5 개선 결과에 실제로 기여하는지 판단.
- A/B 단계는 비교 전용이므로 반드시 `--no-history`를 사용한다.

### Step 6. 다중 시드 검증 (채택 후보 있을 때만)

```bash
python -m features.analysis.scripts.run_multi_seed --seeds 20 --write-result "features/analysis/scripts/연구 분석 결과.md"
```

- 최종 채택 직전에는 필요 시 `--seeds 100` 확대.

### Step 7. 기록

- `연구 분석 결과.md`에 아래를 기록:
  - 날짜 / 대상 세트
  - 세트별 실행 순서(예: 2→4→5)와 중간 채택값
  - 방향별 후보 및 결과
  - 등수 지표 + 평균 매치 수
  - 채택/기각 사유

### Step 8. 최종 반영

- **핵심 규칙: 반영은 Step 8에서만, 1회만 수행**
- 반영 항목:
  1. `jl_service.py` 파라미터
  2. `당첨 이력.md` 갱신 (**최종 1회만 저장 실행**)
  3. `README.md` 동기화
- 실행 규칙:
  - Step 1~7: 비교/검증 전용 실행(`--no-history`)만 허용
  - Step 8: 확정값 반영 실행 1회만 허용
  - 같은 사이클에서 `당첨 이력.md` 저장은 Step 8 단 한 번만 허용

### Step 9. 비효율성 점검 (연구 후 필수)

- 각 방향 실행 후 아래 항목을 체크한다:
  1. **탐색 중복**: 서로 다른 방향이 사실상 같은 파라미터 축을 반복했는가?
  2. **기여도 부족**: 실행 비용 대비 개선 폭이 미미한 단계가 있었는가?
  3. **게이트 누락**: 대상 세트 개선 과정에서 다른 세트 회귀를 놓쳤는가?
  4. **재현성 결함**: seed/전략/필터 조건에 따라 결론이 뒤집히는가?
- 위 항목 중 하나라도 해당되면 `연구 분석 결과.md`에 "비효율 이슈"로 기록한다.
- 동일 이슈가 2회 이상 반복되면 다음 사이클에서 해당 단계를 축소/제거하거나 순서를 조정한다.
- **Step 9 이후 규칙**: Step 9~11은 보고/기획 단계이며, 알고리즘 로직/수치/이력 파일을 변경하지 않는다.

### Step 10. 신규 아이디어 발굴 (연구 후 필수)

- 당일 실행 종료 전, 아래 템플릿으로 **신규 아이디어 3개**를 반드시 작성한다.

| 아이디어 | 가설 | 기대 효과 | 구현 난이도 | 다음 실행 여부 |
|---------|------|----------|------------|---------------|
| A안 | 왜 개선될지 설명 | 상/중/하 | 상/중/하 | 보류/실행 |
| B안 | 왜 개선될지 설명 | 상/중/하 | 상/중/하 | 보류/실행 |
| C안 | 왜 개선될지 설명 | 상/중/하 | 상/중/하 | 보류/실행 |

- 우선 후보 예시:
  - `WHEEL_OFFSET_STEPS` 자동 탐색 (그리드/휴리스틱)
  - 시작번호 전략 혼합 비율 동적화(회차 구간별)
  - 세트 간 커버리지 최대화 목적함수 추가
  - 필터 임계값(합계/홀짝/고저) 자체 최적화
- 채택 규칙:
  - 다음 사이클에는 최소 1개 신규 아이디어를 실제 실험으로 승격한다.
  - 3회 연속 기각된 아이디어는 "폐기 후보"로 분류한다.

### Step 11. 결과 보고 포맷 고정 (세션 재시작 대비)

- 매 실행 종료 후 에이전트는 아래 5개 섹션을 **항상 동일한 순서**로 보고한다.
- 이 규칙은 세션이 종료/재시작되어도 반드시 유지한다.

1. **핵심 결과 요약**
   - best-so-far 파라미터
   - 반영된 최종값
   - 전체 지표 변화(3/4/5등, 합계, 평균 매치 수)
2. **세트 회귀 체크**
   - 대상 세트 외 악화 세트 유무
   - 악화가 있으면 세트 번호와 변화량 명시
3. **비효율 이슈 리스트 (Step 9)**
   - 탐색 중복 / 기여도 부족 / 게이트 누락 / 재현성 결함
4. **신규 아이디어 3개 (Step 10)**
   - 아이디어, 가설, 기대효과, 난이도, 다음 실행 여부
5. **다음 사이클 실험 승격 1개**
   - 바로 다음 실행에서 실제 테스트할 항목 1개 지정

- 위 5개 중 하나라도 누락되면 해당 실행은 "보고 불완전"으로 간주한다.

## 채택/기각 기준

| 우선순위 | 기준 | 판정 |
|---------|------|------|
| 1 | 1등 출현 | 즉시 채택 |
| 2 | 3등 이상 증가 | 채택 |
| 3 | 4등 이상 합계 증가 | 채택 |
| 4 | 4등 이상 동일 + 평균 매치 수 증가 | 채택 |
| 5 | 4등 이상 감소 | 즉시 기각 |
| 6 | 5등만 증가, 상위 등수/매치수 개선 없음 | 기각 |
