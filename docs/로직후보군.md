# 저빈도 번호 선정 로직 후보군

> 현재 시스템은 "기간별 최고 출현 번호를 제외"하는 간접 방식으로 저빈도 번호 풀을 만든다.  
> 아래 6가지 방법은 저빈도 번호를 **직접** 또는 **더 정밀하게** 선정하기 위한 대안 후보다.  
> 각 항목의 도입 여부를 판단할 수 있도록 개요 / 계산 방식 / 장단점 / 현재 구조 호환성을 정리한다.

---

## 방법 2. 표준편차 기반 이상값 탐지 (Z-score)

### 개요
전체 번호(1~45)의 출현 횟수 분포에서 **평균 대비 얼마나 적게 나왔는지** Z-score로 수치화하여 저빈도 번호를 탐지한다.

### 계산 방식
1. 각 번호의 출현 횟수를 집계: `count(n)` for n in 1~45.
2. 전체 평균 `μ`, 표준편차 `σ` 계산.
3. 각 번호의 `z = (count(n) - μ) / σ` 계산.
4. `z < -1.0` (또는 `-1.5`, `-2.0`) 인 번호를 저빈도로 선정.

### 장점
- 전체 분포를 기준으로 **상대적** 저빈도를 정의하므로 기준이 명확하다.
- 임계값(threshold)을 조정해 저빈도 번호 수를 컨트롤할 수 있다.
- 기간 윈도우별로 각각 적용해 현재 구조와 자연스럽게 통합 가능.

### 단점
- 출현 횟수 분포가 정규분포에 가깝다는 가정이 필요하다 (실제로는 근사적으로 성립).
- 현재 로직(최고 출현 1개 제외)보다 제외 번호 수가 가변적이다.
- σ가 작을수록(분포가 평평할수록) 의미 있는 이상값이 적어진다.

### 현재 구조 호환성
- **높음** — 기간 윈도우 구조를 그대로 활용하면서 집계 후 Z-score 계산 추가.
- 백엔드 또는 프론트엔드 어디서나 계산 가능.

### 도입 검토 메모
- 현재 "최고 출현 번호 제외" 로직의 **역방향 버전**으로 대체 또는 병행 가능.
- 임계값을 `-1σ` / `-2σ` 두 단계로 제공하면 사용자가 강도를 선택할 수 있다.

## 방법 4. 가중 점수 집계 (Multi-window Rank Aggregation)

### 개요
현재처럼 기간별로 "1등 번호만 뽑는" 방식 대신, **모든 번호에 기간별 출현 순위 점수를 부여**하고 합산하여 전체적으로 가장 적게 나온 번호를 선정한다.

### 계산 방식
1. 6개 기간 윈도우(전체/10년/5년/3년/1년/6개월) 각각에서 번호별 출현 횟수 집계.
2. 각 기간에서 출현 횟수 역순으로 순위(rank) 부여 (적게 나올수록 낮은 순위).
3. 번호별 6개 기간 순위 합산 → 합산 점수가 낮을수록 저빈도 번호.
4. 하위 N개 번호를 저빈도 후보로 선정.

### 장점
- 현재 6개 기간 윈도우 구조를 **완전히 재활용**한다.
- "어떤 기간에서도 적게 나온 번호"를 균형 있게 선발할 수 있다.
- 가중치를 기간별로 다르게 설정해 최근 기간에 더 높은 가중치 부여 가능.

### 단점
- 단순 합산은 기간별 표본 크기 차이를 무시한다 (전체 기간은 수천 회, 6개월은 26회).
- 가중치 설정이 주관적일 수 있다.

### 현재 구조 호환성
- **매우 높음** — 기존 윈도우 집계 결과에 순위 계산 로직만 추가.
- 기존 `windowTopNumbers` 응답 구조를 확장하거나 별도 필드로 제공 가능.

### 도입 검토 메모
- **현재 구조와 가장 자연스럽게 연결**되는 방법.
- 현재 로직의 약점(기간별 1개만 제외하는 단순함)을 보완하면서 구현 비용이 낮다.
- 기간별 가중치를 `1.0 / sqrt(drawCount)` 형태로 표본 크기 정규화하면 단점 보완 가능.

## 방법 6. 위치별(Position) 빈도 분석

### 개요
로또 번호는 오름차순으로 정렬되므로 `num1`(가장 작은 수) ~ `num6`(가장 큰 수) 위치별로 출현 분포가 다르다. 각 **위치별로** 빈도를 분리하여 특정 위치에서 저빈도인 번호를 분석한다.

### 계산 방식
1. 위치별 집계: `num1`에 나타난 번호 빈도, `num2`에 나타난 번호 빈도, ..., `num6`에 나타난 번호 빈도.
2. 각 위치에서 출현 횟수가 가장 낮은 번호 선정.
3. 세트 생성 시 위치별 저빈도 번호를 우선 배치.

### 장점
- 현재 전체 합산 방식에서 포착하지 못하는 **위치 특이성**을 반영한다.
- 예: 1번 위치에서 "15"가 적게 나왔다면, 전체 합산에서는 묻히더라도 위치 분석으로 발견 가능.

### 단점
- 각 위치의 유효 범위가 다르다 (num1은 주로 1~15, num6은 주로 35~45).
- 위치별 번호가 겹치지 않게 세트를 구성하는 로직이 추가로 필요하다.
- 번호 수가 많지 않아 위치별 표본이 작다 → 통계적 신뢰도 낮음.
- 현재 생성 로직(JL Wheel)과 통합하려면 상당한 구조 변경이 필요하다.

### 현재 구조 호환성
- **중간** — 백엔드 집계 SQL은 위치별로 분리하면 되지만, 프론트엔드 파이프라인과 생성 로직 모두 수정 필요.

### 도입 검토 메모
- 분석·시각화 탭에서 **인사이트 제공용**으로는 유용하지만, 번호 추천 로직 핵심에 넣기에는 구현 비용 대비 효용이 낮을 수 있다.
- 도입한다면 분석 페이지에 위치별 히트맵 차트를 먼저 추가해 데이터를 검증하는 순서가 적절.
