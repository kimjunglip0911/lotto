# 누적 번호 분석 백테스트 리포트 (1차)

## 1. 목적과 한계

- **목적**: 당첨 회차 `D`를 맞히기 직전(`draw_no < D`)까지의 당첨 이력만으로 번호별 누적 출현을 만들고, 그 집계에서 **번호 4개**를 고르는 여러 규칙을 백테스트해 과거 데이터에서 **본번호 6개와의 교집합**이 얼마나 자주 1개 이상 나오는지 비교한다.
- **한계**: 로또는 무작위 추첨에 가깝고, 과거 빈도가 미래 당첨 **확률**을 보장하지 않는다. 본 문서의 비율은 **과거 구간에 대한 경험적 적중률**이며 “기대 당첨 확률”이 아니다.

## 2. 데이터·API·회차 경계

| 항목 | 내용 |
|:---|:---|
| 데이터 소스 | FastAPI `GET /api/analysis/accumulated-numbers/winning-numbers-range?draw_no={max+1}` → DB에서 `draw_no < max+1` 전체 행 |
| 회차 목록 | `GET .../draw-numbers` |
| 회차 경계 | 회차 `D` 분석 시 **항상 `D` 미만**만 사용(백엔드 쿼리와 동일). 100회 당첨을 볼 때는 1~99회만 누적에 포함되는 것과 같다. |
| 백테스트 평가 구간 | 스크립트는 **100회 이상 ~ DB 최신 회차**만 집계한다. 그보다 작은 회차는 직전 누적이 짧아 지표 왜곡이 커서 제외한다. 하한은 `ACCUMULATED_BACKTEST_MIN_DRAW`로 바꿀 수 있다(기본 100). |
| 집계 단위 | 차트와 동일: **본번호 6개 + 보너스 1개** 출현 합을 1~45번에 누적 |
| 적중 단위 | **본번호 6개만** 예측 4개와 교집합 (`toMainNumbersOnly`) |

## 3. 윈도우(기간) 정의

- **UI 고정 윈도우**: 이전 4, 12, 26, 52, 156, 260, 520회차( [`constants.ts`](constants.ts) 의 `WINDOW_CONFIGS`).
- **백테스트 추가 스윕**: 위 크기에 더해 **이전 4~520회차, 8회차 간격**(4, 12, 20, …, 520)으로 후보를 두어, UI에 없는 길이에서도 패턴을 본다. 구현은 [`logic/backtestEngine.ts`](logic/backtestEngine.ts) 의 `getDefaultBacktestWindowSizes()`.

## 4. 번호 4개 선택 전략

### 4.1 본 리포트·스크립트 기본 기준(두 가지)

백테스트 스크립트는 기본으로 아래 **두 전략만** 돌리고, **UI 고정 기간 + 4~520회 8간격 스윕**으로 기간을 바꿔가며 같은 지표를 비교한다. 네 전략을 한꺼번에 보려면 `ACCUMULATED_BACKTEST_STRATEGIES=all`을 지정한다.

| 키 | 설명 |
|:---|:---|
| `nearestMean4` | 45개 번호 출현 평균에 **절대 오차가 가장 작은** 번호 4개(이하 **평균근접**) |
| `twoHotTwoCold` | 출현 상위 2개 + 하위 2개(겹치면 다음 후보로 채움)(이하 **상2+하2**) |

### 4.2 참고용(전체 4종 실행 시)

| 키 | 설명 |
|:---|:---|
| `top4Frequency` | 누적 출현 횟수 **많은** 번호 상위 4개 |
| `bottom4Frequency` | 출현 횟수 **적은** 번호 하위 4개 |

## 5. 백테스트 절차

1. 전체 이력을 한 번 로드한 뒤 메모리에서 회차별로 `prior = draw_no < D`를 자른다.
2. `prior`의 **마지막 N회**만으로 `buildNumberCounts` 집계(N = 윈도우 길이, 부족하면 있는 만큼).
3. 전략으로 번호 4개를 고르고, 해당 `D`회 **실제 본번호 6개**와 교집합 개수를 센다.
4. 회차 **`D >= 100`**(기본)부터 최신까지 반복해 **평균 적중**, **“≥1 적중” 비율**, **최소 적중**, **최대 연속 미적중 길이**를 집계한다.

**실행**: 백엔드 8010 가동 후 `cd frontend && npm run backtest:accumulated-numbers`  
환경 변수:

- `ACCUMULATED_BACKTEST_API_URL` — API 베이스
- `ACCUMULATED_BACKTEST_MIN_DRAW` — 평가 시작 회차(기본 100)
- `ACCUMULATED_BACKTEST_MAX_DRAWS` — 그중 최근 N회만
- `ACCUMULATED_BACKTEST_STRATEGIES` — `focus`(기본, 평균근접+상2하2), `all`(네 전략), 또는 `nearestMean4,twoHotTwoCold` 형태

### 6.6 페이지 고도화 규칙(실시간 선택 회차 기준)

백테스트 테이블(고정 결과)과 별개로, 페이지 검색 시 선택한 회차 `D`에 대해 아래 실시간 규칙으로 최종 4개를 만든다.

1. 전략별 최적 기간 2개 선정  
   - 단기 전략(평균근접): 상위 기간 2개 (기본 short 후보 우선)  
   - 장기 전략(상2+하2): 상위 기간 2개 (기본 long 후보 우선)
2. 각 전략의 **최고 기간 1개**에서 전략별 번호 4개를 산출
3. 두 전략 8개에서 **공통 번호 우선 채택**
4. 4개가 안 차면 전략 점수(백테스트 성능 기반 점수 합)로 상위 번호를 보충

> 화면에는 전략별 추천 4개, 공통 번호, 최종 채택 4개와 함께 보조 지표(`≥1 적중률`, `평균 적중`, `최대 연속 미적중`)를 함께 보여준다.

## 7. 작업지시 항목 대응

| 요구 | 대응 |
|:---|:---|
| 당첨 확률 높은 번호 4개 | 전략별로 4개를 고정 규칙으로 산출·비교(§4). “최고”는 백테스트 지표로만 해석(§1 한계). |
| 백테스트로 형태 비교 | 기본은 **평균근접·상2+하2** + 기간 스윕(§3·§6). 필요 시 `…STRATEGIES=all`로 상·하위 단독 포함. |
| 평균근접/상2+하2 각각 기간 2개 차트 | 페이지에 전략별 상위 2개 기간 차트를 추가해 기존 차트와 함께 비교 가능하게 반영(§6.6). |
| 평균 및 상2+하2 각각 4개 번호 채택 | 전략별 최고 기간 1개 기준으로 각 4개 번호를 산출해 화면에 표시(§6.6). |
| 8개 중 공통 우선 + 확률 높은 번호 보충 | 공통번호 우선 채택 후 전략 점수 합 상위 번호로 최종 4개를 채우는 규칙 적용(§6.6). |
| 매회차 4개 중 1개 이상 | 회차마다 rolling 4개 vs 당첨 6개로 **≥1 비율**을 집계. **매 회차 100% 만족**은 본 데이터·규칙으로는 달성하지 못함(최소 적중 0 존재). |

| 지속 고도화 | 본 MD를 갱신하며 전략·윈도우·지표를 누적 |

## 8. 다음 고도화 아이디어

- **집계 분리**: 번호 선택용 집계는 **본번호만** 카운트하고, 보너스는 별도 지표로 두어 차트용과 예측용을 분리한다.
- **가중 누적**: 최근 회차에 더 큰 가중(지수 감쇠 등)을 두어 cold/hot 정의를 바꾼다.
- **잔차·z-score**: 윈도우 내 기대 출현 대비 잔차가 큰 번호를 고른다.
- **앙상블**: 서로 다른 윈도우에서 뽑은 4개 후보를 투표하거나, 2+2를 서로 다른 N으로 섞는다.
- **평가 하한 조정**: 기본은 `D >= 100`이며, 더 보수적으로 하려면 `ACCUMULATED_BACKTEST_MIN_DRAW`를 올린다.
- **보너스 적중**: 별도 열로 보너스 포함 적중을 기록한다.
